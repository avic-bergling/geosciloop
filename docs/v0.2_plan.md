# GeoSciLoop v0.2 Plan

## v0.2 Goals

- Add a fixture-based real-data adapter prototype for Urban Heat Island workflow planning.
- Keep all required tests offline, deterministic, and free of credentials.
- Represent real-data planning inputs as structured schema objects.
- Generate dry-run artifacts that can be audited without downloading data.
- Validate geospatial metadata for CRS, resolution, alignment, NoData, cloud/shadow QA, provenance, and split strategy.
- Integrate dry-run plans into the research ledger, summary, report, and benchmark evaluator.

## v0.2 Non-Goals

- Do not perform live STAC, GEE, OSM, or population-grid access by default.
- Do not require API keys, Google Earth Engine authentication, or internet access.
- Do not download remote datasets or large files.
- Do not make real-world UHI conclusions.
- Do not claim GeoSciLoop is a complete autonomous scientist.
- Do not add LangGraph, Snakemake, Prefect, CrewAI, AutoGen, or another stateful harness.
- Do not change the v0.1 version number, tag, or release history during v0.2 implementation.

## Target Architecture

v0.2 keeps the simple Python runner. The CLI loads a config, identifies `mode: dry_run`, and dispatches to a dedicated dry-run workflow. Existing v0.1 synthetic configs continue to load through the same schema compatibility layer and run through `run_uhi_workflow`.

The dry-run architecture is:

```text
config -> real-data schema -> adapter registry -> fixture adapters
-> adapter_plan.json -> data_source_manifest.json
-> metadata validators -> ledger -> dry_run_report.md
-> summary.md -> benchmark_summary.json -> run_log.json
```

The runner remains deterministic and file-oriented. Provider-backed access is deferred and must stay outside required tests.

## Data Adapter Design

The adapter layer exposes a common interface:

- `plan(request) -> AdapterPlan`
- `search(request) -> list[dict]`
- `describe(item, request) -> DataSourceRecord`
- `fetch(item, request, dry_run=True) -> FetchResult`

For v0.2:

- STAC is fixture-backed by default.
- GEE is dry-run/stubbed by default.
- OSM is fixture-backed by default.
- Population-grid metadata is fixture-backed.
- Fixture adapters return realistic metadata but do not download data.
- Live adapter calls remain explicit future work.

## Dry-Run Workflow Design

The dry-run workflow for `configs/uhi_real_pilot_template.yaml` will:

1. Parse the dry-run config.
2. Build adapter plans for every requested source.
3. Load fixture metadata from `tests/fixtures/`.
4. Convert fixture items into `DataSourceRecord` entries.
5. Write `adapter_plan.json`.
6. Write `data_source_manifest.json`.
7. Run metadata validators.
8. Write `metadata_validation_report.json` and `validation_report.json`.
9. Write `research_ledger.json`.
10. Write `dry_run_report.md`.
11. Write `summary.md`.
12. Write `benchmark_summary.json`.
13. Write `run_log.json`.

No step authenticates, downloads external data, or calls a live network endpoint.

## Fixture Strategy

Fixtures live under `tests/fixtures/` and are intentionally small JSON files. They simulate source metadata for:

- Landsat Collection 2 Level-2 style LST through STAC.
- Sentinel-2 Level-2A style optical imagery through STAC.
- OpenStreetMap roads through an Overpass-like response.
- Population grid metadata through a manifest fixture.
- GEE image collection metadata through a dry-run stub.

Fixture records must include provider, collection or dataset, role, datetime, bbox, CRS, resolution, NoData metadata, cloud cover where relevant, cloud/shadow QA metadata placeholder, mock href, license, query, and provenance fields.

## Metadata Validation Design

The v0.2 validator layer emits existing `ValidationResult` records with `pass`, `warn`, and `fail` statuses.

Validators cover:

- CRS metadata presence and mixed CRS warnings.
- Geographic CRS warnings for area/distance analysis when no projected target CRS is recorded.
- Raster resolution presence and mixed-resolution warnings.
- Raster alignment and target grid strategy.
- NoData metadata and NoData handling plan.
- Cloud/shadow QA metadata, cloud-cover threshold, and handling plan.
- Source provenance completeness.
- Split strategy and spatial leakage risk.

Warnings are expected for the template when it deliberately uses a random non-spatial split.

## Artifact Output Design

The dry-run output directory is `outputs/uhi_real_pilot_template/`.

Expected artifacts:

- `adapter_plan.json`
- `data_source_manifest.json`
- `metadata_validation_report.json`
- `validation_report.json`
- `research_ledger.json`
- `dry_run_report.md`
- `summary.md`
- `benchmark_summary.json`
- `run_log.json`

Generated runtime outputs remain ignored by git.

## Ledger Integration Plan

The dry-run ledger records:

- Task config.
- AOI and time range.
- Target variable, predictors, and risk indicators.
- Data source requests.
- Adapter plans.
- Data source manifest.
- Metadata validation results and summary.
- Benchmark summary.
- Output artifact paths.
- Report claims, limitations, and assumptions.
- `download_performed: false`.
- `credentials_required: false`.
- `dry_run: true`.

Existing v0.1 ledger behavior remains unchanged.

## Benchmark/Evaluator Integration Plan

Add a dry-run evaluator path with decomposed sub-scores:

- `execution_success`
- `artifact_completeness`
- `data_source_manifest_completeness`
- `metadata_validation_status`
- `provenance_completeness`
- `adapter_plan_completeness`
- `ledger_completeness`
- `reproducibility`
- `report_disclaimer_quality`
- `no_live_dependency_check`

The evaluator remains an engineering and reproducibility quality gate. It does not compute an opaque AI scientist score and does not imply scientific validity.

## Migration Impact From v0.1

- v0.1 configs remain valid and do not need real-data schema fields.
- v0.1 synthetic artifacts keep their existing names and benchmark schema.
- The CLI gains `--dry-run` while preserving `--offline`.
- Optional live adapters remain optional and outside required tests.
- No new required dependencies are introduced.

## Known Limitations

- v0.2 plans real-data workflows but does not execute live remote-sensing or GIS analysis.
- Fixture metadata is representative, not authoritative source metadata.
- No raster pixels, vector geometries, or population grids are downloaded or processed.
- Metadata validators check planning completeness, not scientific correctness of real products.
- Random split warnings are expected until a spatial/block split design is implemented.
- Provider-backed access requires future explicit configuration and separate tests outside the offline baseline.
